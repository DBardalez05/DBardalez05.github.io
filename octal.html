Overview

As part of the OCTAL Student Research Team, I worked on embedded battery-monitoring and protection firmware for a heavy-lift firefighting drone platform. I started by bringing up the sensing paths for cell voltages and thermistors, then expanded the firmware into a more complete BMS loop with current sensing, CAN reporting, fault detection, state transitions, and passive cell balancing. The final handoff included logic to identify imbalanced cells, command ADBMS discharge bits, verify configuration writes through readback, and disable balancing during unsafe conditions.

My Contributions

  • Modified and debugged STM32 firmware for battery-monitoring bring-up on the OCTAL BMS board.
  • Brought up SPI communication with the ADBMS battery-monitoring path and verified register readback during board bring-up.
  • Implemented per-cell voltage readout, pack-voltage calculation, and CAN-scaled voltage reporting for a 14-cell battery pack.
  • Implemented multiplexed thermistor readout using I2C-controlled MAX14661 muxes and STM32 ADC inputs, including voltage-to-temperature conversion and temperature fault handling.
  • Integrated current-sensing readout through the STM32 ADC/OPAMP path and added CAN reporting for pack current.
  • Added BMS fault and state logic for undervoltage, overvoltage, undertemperature, overtemperature, overcurrent, mux faults, PEC faults, current-sense faults, and ADBMS communication faults.
  • Implemented passive cell balancing in a dedicated BALANCING state using ADBMS discharge bits, balancing masks, config readback verification, and safety checks for voltage, temperature, current, and communication faults.
  • Cleaned up CAN signal scaling so cell voltages are reported in 10 mV units, pack voltage in 0.1 V units, temperatures as signed Celsius values, and pack current across the expected operating range.

Technical Highlights

A major part of this work was establishing reliable communication between an STM32 microcontroller and the battery-monitoring path over SPI. Before higher-level BMS behavior could be trusted, I had to verify that register reads were valid, that returned values were physically meaningful, and that the battery-monitor interface behaved consistently on real hardware.

On the voltage-sensing side, I implemented firmware to read per-cell voltages from the battery monitor and calculate overall pack voltage by summing the cell measurements. This made it possible to track both individual cell behavior and full-pack voltage from the firmware layer.

I also implemented the thermistor sensing path by following the hardware architecture from the schematics. The thermistor network is routed through MAX14661 analog multiplexers controlled over I2C, with the selected outputs measured by STM32 ADC pins. I added the mux-control logic, thermistor scanning, voltage conversion, and voltage-to-temperature conversion needed to turn raw ADC values into usable temperature data.

After the sensing paths were validated, I extended the firmware into a broader BMS monitor. The board continuously evaluates minimum and maximum cell voltage and temperature against operating thresholds, sets fault bits when unsafe conditions are detected, and transitions from IDLE to FAULTED when the pack moves outside allowed operating conditions.

I also added passive cell balancing logic to move the firmware beyond only monitoring. The balancing state compares all cell voltages, uses the lowest cell as the reference, and builds a mask for cells that should be bled down through the ADBMS discharge paths. The firmware maps cells 1-12 into CFGRA discharge bits and cells 13-14 into CFGRB discharge bits, writes the config to the battery monitor, and verifies the result through register readback. Balancing is disabled on startup, faults, high current, high temperature, invalid readings, or communication errors.

Challenges

One of the main challenges was bringing up the current-sensing path through the STM32 internal OPAMP and ADC. This was more involved than a direct ADC read: the current-sense signal had to be routed into the STM32, amplified through OPAMP2, sampled through the correct ADC internal channel, and converted into a pack-current value that could be reported over CAN. Debugging required separating possible issues in hardware wiring, OPAMP configuration, ADC channel selection, gain, offset calibration, and board behavior under real pack conditions.

As the expected operating range became clearer, I adjusted the firmware approach for current measurement. An initial 4x OPAMP gain was useful for smaller signals, but it reduced headroom for high-current events. I moved the firmware toward a 2x OPAMP configuration and updated the conversion logic so the BMS could represent pack current more realistically for CAN reporting and overcurrent fault detection.

Passive balancing introduced a different kind of risk because the firmware was no longer only reading sensor values; it was commanding discharge paths on the battery monitor. To reduce the chance of bleeding the wrong cell, I structured balancing around a dedicated BALANCING state, conservative entry and exit checks, mask-based cell selection, ADBMS configuration writes, and readback verification before treating a balance command as successful.
