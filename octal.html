Overview

As part of the OCTAL Student Research Team, I work on embedded systems tasks related to battery monitoring and sensor bring-up for a heavy-lift firefighting drone project. My role has focused on making the STM32 communicate reliably with battery-monitoring hardware and sensing circuitry, then validating that the returned data reflects real physical measurements rather than just successful communication.

This work has involved firmware bring-up, hardware-aware debugging, and measurement validation across multiple interfaces. Rather than treating sensing as isolated code, I have worked at the boundary between firmware and hardware to establish reliable readout paths for both cell voltages and thermistors.

My Contributions

  • Modified and debugged STM32 firmware for battery-monitoring and sensing bring-up.
  • Brought up SPI communication with the ADBMS battery-monitoring path and verified valid register readback behavior.
  • Implemented and tested per-cell voltage readout using battery-monitor IC register data over SPI.
  • Determined the thermistor signal path from schematic review and implemented multiplexed thermistor readout using STM32 I2C-controlled MAX14661 muxes and local ADC measurement.
  • Mapped thermistor channels from hardware schematics into firmware and validated successful thermistor voltage acquisition.
  • Supported CAN communication integration with the broader drone electrical system.
  • Worked through bring-up issues by debugging firmware sequencing, communication behavior, and hardware-state assumptions together.

Technical Highlights

A major part of my work has been establishing reliable communication between an STM32 microcontroller and the battery-monitoring path over SPI. This included bringing up the firmware interface, validating register read behavior, and confirming that the communication path was functioning correctly before trusting higher-level sensing data.

On the voltage-sensing side, I implemented per-cell voltage readout through the ADBMS interface. Instead of relying on direct MCU analog measurement, the STM32 requests and parses battery-monitor register data over SPI, which more closely reflects real embedded battery-management bring-up.

I also implemented the thermistor readout path by tracing the sensing architecture from the board schematics and separating it from the battery-monitor path. The thermistor network is routed through MAX14661 analog multiplexers controlled over I2C, with the selected outputs read locally by STM32 ADC pins. I added firmware to configure the muxes, scan thermistor channels, and acquire validated thermistor voltage data.

This required understanding how multiple communication and measurement paths coexist on the same board: SPI for battery-monitor communication, I2C for thermistor mux control, and ADC for analog thermistor measurement. Working across those interfaces gave me hands-on experience with real hardware/firmware integration rather than isolated coding.

Challenges

The hardest part of the project has been debugging at the firmware-hardware boundary. During bring-up, incorrect behavior can come from SPI configuration, I2C device addressing, mux selection logic, ADC interpretation, board state, or missing physical inputs, so each issue has to be narrowed down carefully rather than treated as a simple code bug.

Another challenge has been separating “communication works” from “measurement is valid.” For both voltage and thermistor sensing, it was important to confirm not only that the interface responded, but that the returned values matched the actual hardware path and expected electrical behavior.
